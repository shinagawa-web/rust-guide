# テストを追加する

前のページで `file_type` を直しました。ただ、このまま終えると、あとでコードを変えたときに同じ間違いが再び入り込んでも気づけません。ディレクトリを渡すと先頭に `d` が付く、という直した動きをテストにして、あとから壊れないようにします。

`perm.rs` の末尾には、もともとテストが書かれています。前のページまで読むのを保留していた部分です。まずそれを読み、書き方をつかんでから、ディレクトリのテストを足します。

## 既存のテストを読む

`perm.rs` の末尾はこうなっています。

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn formats_regular_file() {
        // 0o100755: 通常ファイルで rwxr-xr-x
        assert_eq!(format_mode(0o100755), "-rwxr-xr-x");
    }
}
```

`#[cfg(test)]` は、この下のブロックをテストのときだけコンパイルする目印です。`cargo test` のときだけ使われ、`cargo run` では無視されます。`mod tests` はテストをまとめるモジュールで、`use super::*;` はその外側にある `format_mode` などをこの中から呼べるようにする 1 行です。

テスト 1 つは `#[test]` の付いた関数です。中の `assert_eq!` は、2 つの値が等しいことを確かめるマクロで、等しければ成功、違えばその場で失敗します。この `formats_regular_file` は、通常ファイルの `0o100755` を `format_mode` に渡した結果が `-rwxr-xr-x` になることを確かめています。

`0o100755` は、上の桁 `0o100000` が通常ファイルの種別、下の `755` が権限です。だから期待する結果は、先頭が `-` の `-rwxr-xr-x` になります。

## ディレクトリのテストを足す

同じ書き方で、ディレクトリのテストを足します。ディレクトリの種別は `0o040000` なので、権限が `755` なら `0o040755` です。これを渡した結果が `drwxr-xr-x`（先頭が `d`）になることを確かめます。

```rust
    #[test]
    fn formats_directory() {
        // 0o040755: ディレクトリで rwxr-xr-x
        assert_eq!(format_mode(0o040755), "drwxr-xr-x");
    }
```

`mod tests` の中、`formats_regular_file` の下に足します。

## テストを実行する

`cargo test` で実行します。

```sh
$ cargo test
```

```text
running 2 tests
test perm::tests::formats_regular_file ... ok
test perm::tests::formats_directory ... ok

test result: ok. 2 passed; 0 failed
```

2 つのテストがどちらも通りました。もとからあった通常ファイルのテストと、いま足したディレクトリのテストです。前のページの修正が、既存のテストを壊さずにディレクトリにも対応できていることが確かめられました。

## テストが修正を守る

もし前のページの修正をしていなかったら、いま足したテストは通りません。`file_type` が `-` を返したままなので、結果が `-rwxr-xr-x` になり、期待した `drwxr-xr-x` と食い違って失敗します。

```text
assertion `left == right` failed
  left: "-rwxr-xr-x"
 right: "drwxr-xr-x"
```

このテストがあれば、あとから `file_type` が元の形に戻ってしまっても、`cargo test` で失敗としてすぐ気づけます。直した動きが元に戻っていないかを確かめられる、これがテストを足す理由です。

これで、最初に決めた 2 つの条件——修正後も既存のテストが通ることと、ディレクトリのテストを追加すること——の両方を満たせました。次のページでは、ここまで何をどう変えたのかをふり返り、自分の言葉で説明できるようにして、この初級編を終えます。
