# 作ったツールを人に渡す

初級で直した fmode は、手元では動きます。ただ、動かす方法はまだ `cargo run` だけです。プロジェクトのディレクトリまで移動して `cargo run` と打つ。開発中はそれでよかったのですが、日常で使うには手間がかかります。どこにいても `fmode` と打つだけで動いてほしいし、同じものを他の人にも配りたくなります。

この章では、fmode を自分の環境に入れて、他の人へも配れるようにします。完璧に仕上げてからではなく、動いた今のうちに配る、という進め方です。

## 自分の環境に入れる

どこからでも使えるようにするのが、いちばん早い配布先です。プロジェクトのディレクトリで次のコマンドを実行します。

```sh
$ cargo install --path .
```

`cargo install` は、リリース向けにビルドした実行ファイルを `~/.cargo/bin` という決まった場所にコピーするコマンドです。`--path .` は「今いるディレクトリのプロジェクトを対象にする」という指定です。

本当にコピーされたか、置き場所を覗いて確かめます。

```sh
$ ls ~/.cargo/bin/fmode
```

```text
/home/あなた/.cargo/bin/fmode
```

パスがそのまま表示されれば置けています。`No such file or directory` と出たら、ビルドかインストールのどこかで失敗しているので、`cargo install --path .` の出力を見直します。

`~/.cargo/bin` は、Rust を入れたときに PATH（コマンドを探す場所の一覧）へ登録されています。だから、ここに入った実行ファイルは名前だけで呼び出せます。`cargo run` と違って、プロジェクトのディレクトリにいる必要はありません。

念のため、プロジェクトの外からでも呼び出せるか確かめます。

```sh
$ cd ~
$ fmode /etc/hosts
```

```text
-rw-r--r--  /etc/hosts
```

`fmode: command not found` と出たときは、`~/.cargo/bin` に PATH が通っていません。ターミナルを開き直すか、`source ~/.cargo/env` を実行してからもう一度試します。それでも直らなければ、Rust のインストールをやり直してください。

実行ファイルの名前が `fmode` になるのは、`Cargo.toml` の `name` から来ています。

```toml
[package]
name = "fmode"
```

コードを直したら、もう一度 `cargo install --path .` を実行すれば、入っている fmode が新しいものに置き換わります。

## 配布の準備をする

ここから先は、自分のマシンの外へ配布する話です。fmode が自分の GitHub リポジトリに置いてあることを前提にします。

出発点は他人のリポジトリを clone したものなので、`git push` の送り先（origin）はそのリポジトリを指したままです。まず GitHub で自分のアカウントに空のリポジトリを作り、送り先を自分のものへ張り替えてから push します。

```sh
$ git remote set-url origin https://github.com/あなたのアカウント/fmode.git
$ git push -u origin main
```

送り先を張り替えられたら、配る準備は完了です。

## 実行ファイルを配る

他の人に配るときは、ビルド済みの実行ファイルそのものを配布します。相手は Rust を用意する必要はありません。ソースからビルドしてもらうのではなく、できあがったファイルを受け取って置くだけ、という形です。

まず、配るための実行ファイルを用意します。ここで初めて `cargo build` を使います。

これまで動かすのに使ってきた `cargo run` は、ソースコードを実行ファイルに変換（ビルド）してから、それを走らせるコマンドでした。配るのに要るのは前半のビルドだけです。走らせずに止めて、できあがった実行ファイルそのものを取り出せばいい。それをやるのが `cargo build` です。

```sh
$ cargo build --release
```

`--release` を付けると、最適化した実行ファイルが `target/release/fmode` にできます。`cargo run` が使う開発用のビルドより動作が速く、配るのはこちらです。

このファイルを GitHub の Release に添付します。`gh release create` コマンドを使うと、タグの作成・リリースの公開・ファイルの添付を一度に行えます。

```sh
$ gh release create v0.1.0 target/release/fmode
```

`v0.1.0` がリリースのタグになります。コマンドを実行するとリリースページの URL が表示されるので、そこから添付されたことを確認できます。

受け取る人は、その 1 ファイルをダウンロードして、実行できる状態にするだけです。

```sh
$ chmod +x fmode
$ ./fmode /etc/hosts
```

`chmod +x` は、ダウンロードしたファイルを実行できる状態にするコマンドです。

macOS では、ブラウザでダウンロードした実行ファイルに安全確認のための印が付き、最初の実行がブロックされることがあります。その場合は `xattr -d com.apple.quarantine fmode` で印を外すか、システム設定の「プライバシーとセキュリティ」から実行を許可します。

一つ注意があります。ビルドした実行ファイルは、ビルドした環境と同じ種類の OS でしか動きません。macOS で作ったものは macOS 向け、というように分かれ、同じ OS でも CPU の種類が違うと動かないことがあります。複数の環境向けをまとめて用意する方法もありますが、この章では踏み込みません。まずは自分と同じ環境の人に配布するところから始めれば十分です。

## 動かして確かめる

プロジェクトの外から呼んでみます。

```sh
$ cd ~
$ fmode
```

```text
-rw-r--r--  .bashrc
drwxr-xr-x  Documents
```

`cargo run` なしで、どこからでも fmode が呼び出せるようになりました。

配れる形にはなりましたが、`ls -al` の出力にはリンク数・所有者・グループ・サイズ・更新日時も並んでいます。次の章では、OS がファイルごとに持っている生のデータ（stat 構造体のフィールド）にたどり着く方法を見ていきます。
