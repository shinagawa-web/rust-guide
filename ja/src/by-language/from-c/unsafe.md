# 未定義動作と unsafe・FFI

C言語 には未定義動作があります。ヌルポインタの参照、範囲外アクセス、解放済みメモリの読み書き。こうした操作をしたとき、C の標準は何も保証しません。Rust はこれらの多くをコンパイル時に防ぎますが、どうしても必要なときのために `unsafe` という逃げ道を用意しています。この章では、C の未定義動作と Rust の `unsafe` の関係を見ます。

## C の未定義動作

これまでの章でいくつか見てきました。ヌルポインタの参照（4章）、範囲外アクセス（3章）、タグと値の取り違え（7章）。C の標準ではこれらを「未定義動作」と呼び、何が起きても規格上は正しい動作とされます。クラッシュすることもあれば、何事もなく続くこともあり、コンパイラの最適化によって全く別の動作になることもあります。

```c
// C — いずれも未定義動作
int *p = NULL;
*p = 1;          // ヌルポインタへの書き込み

int xs[3] = {1, 2, 3};
xs[5] = 99;      // 範囲外への書き込み

int *q = malloc(sizeof(int));
free(q);
*q = 1;          // 解放済みメモリへの書き込み
```

## Rust が防ぐもの

Rust は上の三つをコンパイル時または実行時に防ぎます。

- 参照は必ず生きた値を指す（`&T` に NULL はない）
- 範囲外アクセスは実行時にパニックして止まる
- 解放済みメモリへのアクセスは所有権の規則でコンパイル時にはじかれる

```rust
// Rust
fn main() {
    let xs = [1, 2, 3];
    println!("{}", xs[5]); // パニック: index out of bounds
}
```

未定義動作ではなく、「何が起きたか」が明確な停止になります。

## unsafe — 必要なときだけ開ける逃げ道

それでも、生のポインタを扱わなければならない場面はあります。C のライブラリを呼ぶとき、OS のシステムコールを直接叩くとき、パフォーマンス上の理由でチェックを外したいときです。Rust はそのために `unsafe` ブロックを用意しています。

Rust のポインタには参照（`&T`）と生ポインタ（`*const T`）の2種類があります。参照はコンパイラが安全を保証しますが、生ポインタは C のポインタと同じで、NULL かもしれないし、指す先が生きているかどうかもコンパイラは関知しません。生ポインタを参照外しするには `unsafe` ブロックが必要です。

```rust
// Rust
fn main() {
    let x = 42;
    let p = &x as *const i32; // 生ポインタを作る（C の int *p = &x と同じ）

    unsafe {
        println!("{}", *p); // 42 — unsafe ブロックの中でしか参照外しできない
    }
}
```

`unsafe` ブロックの中では、生ポインタの参照外しなど、通常は禁止されている操作ができます。安全かどうかの判断はプログラマに戻りますが、その範囲が `unsafe {}` の中だけに限定されます。C では全体が「unsafe」でしたが、Rust では `unsafe` な操作をどこで行っているかが一目でわかります。

## C のライブラリを呼ぶ — FFI

C で書かれた既存のライブラリを Rust から呼ぶことができます。これを FFI（Foreign Function Interface）と呼びます。

```rust
// Rust
use std::os::raw::c_char;

extern "C" {
    fn strlen(s: *const c_char) -> usize;
}

fn main() {
    let s = c"hello";
    let len = unsafe { strlen(s.as_ptr()) };
    println!("{len}"); // 5
}
```

`extern "C"` で C の関数の型を宣言し、`unsafe` ブロックの中で呼びます。C 側の関数が何をするかは Rust のコンパイラには知る方法がないため、呼び出し全体が `unsafe` になります。

---

C の未定義動作は、何が起きるかわからないという形でプログラム全体に潜んでいました。Rust は安全な操作をデフォルトにし、どうしても必要なときだけ `unsafe` ブロックに明示して許可する設計になっています。次の章では、`pthread` とは異なるアプローチで、データ競合をコンパイル時に防ぐ並行性を見ます。
