# C から Rust へ

C を書いてきた人にとって、Rust は「**C のつもりで書ける低レベル言語**」です。GC もランタイムもなく、出てくる機械語のイメージが頭に浮かぶ感覚はそのまま使えます。違うのは、C で自分の注意力に頼っていた部分（解放忘れ、二重解放、NULL 参照、バッファ外アクセス）を、**コンパイラが型で潰してくれる**点です。

ここでは C の知識を起点に、差分だけを押さえます。

---

## まず結論：対応表

| C でやっていたこと | Rust では |
|---|---|
| `malloc` / `free` | 所有権 + スコープ離脱時の自動 `Drop`（手動 free なし） |
| 生ポインタ `T*` | 参照 `&T` / `&mut T`（借用）。生ポインタ `*const T` は `unsafe` 内のみ |
| `NULL` | `Option<T>`（`None` / `Some(x)`） |
| 戻り値のエラーコード / `errno` | `Result<T, E>` + `?` 演算子 |
| `char*` 文字列 | `&str`（借用）/ `String`（所有）。常に長さを持つ |
| 配列 + 長さの引き回し | スライス `&[T]`（ポインタ＋長さがセット、境界チェックあり） |
| `#define` 定数 | `const` / `static` |
| 関数マクロ | `macro_rules!`（衛生的マクロ）または `const fn` |
| `struct` + 関数群 | `struct` + `impl`（メソッド） |
| ヘッダ / `#include` | モジュール `mod` + `use` |
| `Makefile` / 手動リンク | `Cargo`（ビルド・依存解決・テストを統合） |
| 未定義動作 (UB) | safe Rust では原理的に発生しない |

---

## 1. メモリ管理：free を書かない

C:

```c
char *buf = malloc(256);
if (!buf) return -1;
// ... 使う ...
free(buf);   // 忘れるとリーク、2回呼ぶとクラッシュ
```

Rust:

```rust
let buf = vec![0u8; 256];   // 確保
// ... 使う ...
// スコープを抜けたら自動で解放される。free は書かない。
```

所有権を持つ変数がスコープを抜けると、コンパイラが解放コード（`Drop`）を挿入します。**いつ解放されるかはコンパイル時に確定**していて、GC のような実行時の停止はありません。RAII を言語レベルで強制している、と捉えると正確です。

---

## 2. ポインタ → 参照と借用

C のポインタは「アドレス」でしかなく、有効か・誰が所有しているかは表現できません。Rust は参照に**借用規則**を課します。

```rust
fn parse(data: &[u8]) -> u32 {   // data を「借りる」だけ。所有権は奪わない
    data[0] as u32
}
```

- 同時に存在できるのは「**複数の不変参照** `&T`」**か**「**1つの可変参照** `&mut T`」のどちらか一方
- これにより**データ競合とダングリングポインタがコンパイル時に排除**される

どうしても生ポインタが要る場面（MMIO、FFI、自前アロケータなど）は `unsafe` ブロックで明示します。C のポインタ操作そのものは `unsafe` の中でほぼそのまま書けます。

---

## 3. NULL → Option

C の「失敗したら NULL を返す」は、参照外しのバグの温床です。Rust は値が無いことを型で表します。

```rust
fn find(slice: &[i32], target: i32) -> Option<usize> {
    for (i, &v) in slice.iter().enumerate() {
        if v == target {
            return Some(i);
        }
    }
    None
}

match find(&data, 42) {
    Some(i) => println!("index {i}"),
    None    => println!("not found"),
}
```

`Option` を中身を取り出さずに使うことはできないので、「NULL チェック忘れ」が構造的に起きません。

---

## 4. エラーコード → Result と `?`

C の `if (ret < 0) goto fail;` の連鎖は `Result` + `?` に置き換わります。

```rust
fn read_config(path: &str) -> Result<Config, std::io::Error> {
    let text = std::fs::read_to_string(path)?;  // 失敗時はここで早期 return
    let cfg  = parse(&text)?;
    Ok(cfg)
}
```

`?` は「エラーなら呼び出し元へ返す」という C の `goto fail;` パターンを 1 文字で書けるものだと思ってください。**エラーを無視するとコンパイラが警告**します。

---

## 5. 配列・バッファ → スライス（境界チェック付き）

C で `(ptr, len)` を別々に引き回していたものが、Rust では `&[T]` 1 つにまとまります。長さが型に同梱されるので、引数の取り違えが起きません。

```c
// C: 長さを別に渡す。範囲外アクセスは検出されない
void process(const uint8_t *buf, size_t len);
```

```rust
// Rust: スライスが長さを持つ。範囲外アクセスは panic で止まる（暴走しない）
fn process(buf: &[u8]) { /* ... */ }
```

性能が要る内側ループで境界チェックを外したい場合は、イテレータを使う（最適化で消える）か `get_unchecked`（`unsafe`）を使います。

---

## 6. 組み込み・ベアメタル（no_std）

OS なし・標準ライブラリなしの環境では `#![no_std]` を使います。ヒープすら無い構成も組めます。

```rust
#![no_std]
#![no_main]
```

- `core` クレート … `std` 抜きの言語コア（`Option`, `Result`, スライス等は使える）
- `embedded-hal` … MCU 周辺機器（GPIO/SPI/I2C 等）の共通トレイト。ベンダ非依存でドライバが書ける
- `cortex-m` / `riscv` 等 … アーキ別のサポート

C で書いていたレジスタ直叩きは、`svd2rust` が生成する型付き API か、`unsafe` のポインタ操作で書けます。「所有権でペリフェラルの二重利用を防ぐ」といった、C では手作業だった規律も型で表現できます。

---

## 7. 既存の C 資産との連携（FFI）

全部を一気に書き直す必要はありません。Rust は **C ABI と直接相互運用**できます。

Rust から C を呼ぶ:

```rust
extern "C" {
    fn c_compute(x: i32) -> i32;
}

let y = unsafe { c_compute(10) };
```

C から Rust を呼ぶ（Rust 側を `#[no_mangle] extern "C"` で公開）:

```rust
#[no_mangle]
pub extern "C" fn rust_add(a: i32, b: i32) -> i32 {
    a + b
}
```

ホットスポットや新規モジュールだけ Rust に置き換えて、既存 C と混在させる、という段階的移行が現実的です。

---

## C 経験者がつまずきやすい点

- **「コピーされたと思ったら所有権が移動していた」** — 非 `Copy` 型の代入はムーブ。元の変数は使えなくなる。`Copy` なのは整数等の小さな値だけ。
- **ライフタイム注釈 `'a`** — 「この参照は元データより長生きしない」をコンパイラに伝えるもの。C には対応概念がなく、最初は戸惑うが、ダングリング参照を消すための仕組み。
- **「とりあえずポインタで」が効かない** — 借用規則に合わせて設計する必要がある。最初は窮屈だが、これが UB ゼロの正体。
- **可変グローバル変数** — `static mut` は `unsafe`。代わりに `OnceCell` / `Atomic*` / `Mutex` を使う。

---

## まとめ

C で培った「ハードに近い感覚・予測可能なコスト」はそのまま活かせます。Rust が肩代わりするのは、**C で人間の注意力に頼っていた安全性の部分**です。最初は借用チェッカが厳しく感じますが、それは「動くけど危ういコード」をコンパイル時点で弾いているだけ、と分かると見え方が変わります。
