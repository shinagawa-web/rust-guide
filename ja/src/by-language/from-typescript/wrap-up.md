# TypeScript から Rust へ、ここまで

ここまで、TypeScript から Rust への違いを順に見てきました。まず[TypeScript とほぼ同じところ](basics.md)で、変数・制御フロー・関数・構造体・Vec / HashMap を一気に確認しました。書き方は違っても、考え方は TypeScript とほとんど同じでした。そのあと、同じ感覚では書けなかった数か所を、一つずつ見てきました。

振り返ると、Rust の多くは TypeScript の知識の上にそのまま乗り、引っかかるのは次の数か所だけでした。

- [所有権と借用](ownership.md) — GC の代わりに、コンパイル時にメモリを管理する
- [null / undefined の代わりに Option](option.md) — 「無いかもしれない」を型で表す
- [エラーの扱い](result.md) — `throw` / `try-catch` から、Result と `?` へ
- [構造的な型の代わりに trait](trait.md) — 共通の振る舞いを表す
- [union 型の代わりに enum と match](enum.md) — 種類の違う値を型で分け、扱い漏れを防ぐ
- [並行性](concurrency.md) — 本当に同時に動かしても、競合をコンパイル時に止める

これらは別々の機能に見えて、根はつながっています。とくに所有権は、メモリの管理でも、複数スレッドでのやり取りでも、同じ仕組みになって効いてきます。値を誰が持ち、いつ手放すか。Rust がうるさく見える場面の多くは、この所有権が「今それは安全でない」と教えているだけです。

TypeScript で身につけた型の感覚は、そのまま使えます。「失敗するかもしれない」「無いかもしれない」を型に出し、使う前にコンパイラへ確かめさせる、という発想は、Rust とよく似ています。TypeScript では省くこともできた確認を、Rust は必ず通させる。その徹底の上に所有権という一点を重ねれば、Rust はもう遠い言語ではありません。
