# エラーの扱い

Go でエラーは値です。関数は `(値, err)` を返し、呼ぶ側が `if err != nil` で確かめます。この「エラーを戻り値で運ぶ」という考え方は、Rust もまったく同じです。例外で大域脱出するのではなく、成否を値として返します。

違うのは、その成否を Rust では `Result` という型に載せることです。中身を使う前に成否を確かめることを、コンパイラが要求します。Rust が「無いかもしれない」を `Option` という型で表すのと、そっくり同じ発想です。この章では、文字列を数値に変換する小さな例を Go と Rust で並べ、確認し忘れが Rust ではどう塞がれるかを見ます。

## Go ではエラーを返すが、確認は書く側しだい

文字列を整数に変換する処理を考えます。Go では変換結果とエラーの二つが返り、失敗したかどうかは `err` で分かります。

```go
// Go
n, err := strconv.Atoi("42")
if err != nil {
    fmt.Println("変換できない")
    return
}
fmt.Println(n) // 42
```

ここで `err` を受け取ること自体は Go に強制されますが、`_` で捨てることはできます。捨てると、失敗時にはゼロ値の `0` がそのまま後ろへ流れます。

```go
// Go
n, _ := strconv.Atoi("abc") // 失敗しても err を捨てる
fmt.Println(n)              // 0
```

返ってきた `0` が「変換した結果が 0」なのか「変換に失敗した」のかは、この値だけでは区別できません。`err` を確かめるかどうかは書く側の注意力に委ねられていて、捨ててもコンパイルは通ります。「失敗したかもしれない」ことも、それを確かめたかどうかも、`n` の型には現れません。

## Rust では成否が型に出る（Result）

同じ変換を Rust で書きます。`parse` の戻り値は、数値そのものではなく `Result` です。

```rust
// Rust
fn main() {
    let n = "42".parse::<i32>(); // n の型は Result<i32, ParseIntError>
    println!("{n:?}");           // Ok(42)
}
```

`Result<i32, ParseIntError>` は、次の二つのどちらかです。

- `Ok(42)` — 成功（変換できた）
- `Err(...)` — 失敗（理由が入っている）

`"abc"` を渡せば `Err` になります。Go のゼロ値と違い、失敗は `0` ではなく `Err` として返るので、「結果が 0」と「変換に失敗」が混ざりません。そして `n` は数値そのものではないので、いきなり数として使うことはできません。

```rust
// Rust
fn main() {
    let n = "abc".parse::<i32>();
    println!("{}", n + 1); // コンパイルエラー：Result は i32 のように足せない
}
```

Go では失敗しても `0` が返り、そのまま計算に混ざりました。Rust は、`Ok` か `Err` かを先に確かめない限り、中の数値に触らせません。この確認は任意ではなく、省くとコンパイルエラーになります。

## 成否で分ける — match

`Result` から中身を取り出す基本は、`Option` のときと同じ `match` です。`Ok` と `Err` を分けて処理する形で、Go の `if err != nil { } else { }` にそのまま対応します。

```rust
// Rust
fn main() {
    match "42".parse::<i32>() {
        Ok(n) => println!("{n}"),
        Err(e) => println!("変換できない: {e}"),
    }
}
```

`Err` の側を書かないとコンパイルが通らないので、「失敗した場合」の考慮漏れが起きません。Go の `if err != nil` は書き忘れても通りましたが、こちらは書き漏らせない形です。

`unwrap()` や `expect()` は、確認を省いて中身をそのまま取り出します。ただし `Err` のときは panic します。「ここは絶対に成功するはず」と言い切れる場面に限る最終手段で、`unwrap` という名前がコードに残るので、どこで確認を飛ばしたかが読めば分かります。

```rust
// Rust
fn main() {
    let n = "42".parse::<i32>().unwrap(); // "42" は成功するので 42。"abc" だと panic
    println!("{n}"); // 42
}
```

失敗時に panic させず、既定値で進めたいなら `unwrap_or` を使います。`unwrap` の、`Err` のときに既定値を返す版です。

```rust
// Rust
fn main() {
    let n = "abc".parse::<i32>().unwrap_or(0); // 失敗なら 0
    println!("{n}"); // 0
}
```

## エラーを呼び出し元へ渡す — `?`

Go では、受け取ったエラーをそのまま上へ返す `if err != nil { return err }` を、あちこちで書きます。Rust はこの定型を `?` の一文字にまとめます。`Ok` なら中身を取り出して先へ進み、`Err` ならその場で `Err` を返して関数を抜けます。

```rust
// Rust
use std::num::ParseIntError;

fn parse_sum(a: &str, b: &str) -> Result<i32, ParseIntError> {
    let x = a.parse::<i32>()?; // 失敗したら、ここで Err を返す
    let y = b.parse::<i32>()?;
    Ok(x + y)
}

fn main() {
    println!("{:?}", parse_sum("3", "4"));   // Ok(7)
    println!("{:?}", parse_sum("3", "abc")); // Err(ParseIntError { .. })
}
```

`?` を使うには、関数自身が `Result` を返している必要があります。`Err` をその場で返す先が要るからです。この `?` は `Option` に対しても同じ形で使え、`Option` なら `None` を、`Result` なら `Err` を、それぞれ呼び出し元へ渡します。「無いかもしれない」と「失敗したかもしれない」を、同じ書き味で扱えるようにしてあるわけです。
