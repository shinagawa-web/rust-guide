# エラーの扱い

Go でエラーは値です。関数は `(値, err)` を返し、呼ぶ側が `if err != nil` で確かめます。この「エラーを戻り値で運ぶ」という考え方は、Rust もまったく同じです。例外で大域脱出するのではなく、成否を値として返します。

違うのは、その成否を Rust では `Result` という型に載せることです。中身を使う前に成否を確かめることを、コンパイラが要求します。Rust が「無いかもしれない」を `Option` という型で表すのと、そっくり同じ発想です。

この章は、注文フォームから来た「個数」の文字列を数値にして使う、という場面で進めます。利用者が打つ文字列は数字とは限りません。数字でなかったときにどう振る舞うかが、Go と Rust で分かれます。

## Go では確認を書き忘れても通る

入力の文字列を整数に変換します。Go では変換結果とエラーの二つが返り、失敗したかどうかは `err` で分かります。

```go
// Go
qty, err := strconv.Atoi(input)
if err != nil {
    fmt.Println("個数は数字で入力してください")
    return
}
fmt.Printf("%d 個を処理します\n", qty)
```

ここで `err` を受け取ること自体は Go に強制されますが、`_` で捨てることはできます。捨てると、数字でなかったときにはゼロ値の `0` がそのまま後ろへ流れます。

```go
// Go
qty, _ := strconv.Atoi("abc") // 数字でなくても 0
process(qty)                  // 0 個として処理が進んでしまう
```

`0` が「0 個と入力された」のか「そもそも数字でなかった」のかは、この値だけでは区別できません。`err` を確かめるかどうかは書く側の注意力に委ねられていて、捨ててもコンパイルは通ります。「失敗したかもしれない」ことも、それを確かめたかどうかも、`qty` の型には現れません。

## Rust では成否が型に出る（Result）

同じ変換を Rust で書きます。`parse` の戻り値は、数値そのものではなく `Result` です。

```rust
// Rust
fn main() {
    let qty = "abc".parse::<i32>(); // qty の型は Result<i32, ParseIntError>
    println!("{qty:?}");            // Err(ParseIntError { .. })
}
```

`Result<i32, ParseIntError>` は、次の二つのどちらかです。

- `Ok(3)` — 成功（数値に変換できた）
- `Err(...)` — 失敗（理由が入っている）

数字でない文字列は、Go のように `0` にはならず `Err` になります。「0 個」と「数字でない」が混ざりません。そして `qty` は数値そのものではないので、いきなり数として使うことはできません。

```rust
// Rust
fn main() {
    let qty = "abc".parse::<i32>();
    println!("{}", qty + 1); // コンパイルエラー：Result は i32 のように足せない
}
```

Go では数字でなくても `0` が返り、その `0` を本物の個数と見分けられないまま計算に使ってしまいました。Rust は、`Ok` か `Err` かを先に確かめない限り、中の数値に触らせません。この確認は任意ではなく、省くとコンパイルエラーになります。

## 数字だったときと、そうでないときを分ける — match

`Result` から中身を取り出す基本は、`Option` のときと同じ `match` です。`Ok` と `Err` を分けて処理する形で、Go の `if err != nil { } else { }` にそのまま対応します。数字なら処理を進め、数字でなければ利用者に伝えます。

```rust
// Rust
fn main() {
    let input = "3";
    match input.parse::<i32>() {
        Ok(qty) => println!("{qty} 個を処理します"),
        Err(_) => println!("個数は数字で入力してください"),
    }
}
```

`Err` の側を書かないとコンパイルが通らないので、「数字でなかった場合」の考慮漏れが起きません。Go の `if err != nil` は書き忘れても通りましたが、こちらは書き漏らせない形です。

入力ではなく、自分でコードに書いたリテラルを数値にするときは、失敗しないと分かっています。そういう場面では `unwrap()` や `expect()` で、確認を省いて中身をそのまま取り出せます。ただし `Err` のときは panic します。

```rust
// Rust
fn main() {
    let max = "100".parse::<i32>().unwrap(); // 自分で書いた "100" なので確実
    println!("上限は {max} 個");
}
```

`unwrap` は「ここは絶対に成功するはず」と言い切れる場面に限る最終手段で、利用者の入力のように失敗しうる値には使いません。`unwrap` という名前がコードに残るので、どこで確認を飛ばしたかは読めば分かります。

数字でなかったときに、既定の個数で進めてよいと決めているなら、`unwrap_or` が使えます。

```rust
// Rust
fn main() {
    let qty = "".parse::<i32>().unwrap_or(1); // 未入力なら 1 個
    println!("{qty} 個を処理します");         // 1 個を処理します
}
```

Go はゼロ値で黙って `0` に寄せていましたが、こちらは「数字でなければ 1 個にする」と決めたことがコードに残ります。潰すかどうかも、いくつに潰すかも、書いた側の選択として見えます。

## 失敗を呼び出し元へ渡す — `?`

注文の小計は、個数と単価の二つを数値にして掛けた値です。どちらかが数字でなければ小計は出せません。この「二つとも変換し、片方でも失敗したら中断する」処理を素直に書きたい。

Go なら、変換のたびに `if err != nil { return ..., err }` を挟みます。

```go
// Go
func subtotal(qty, price string) (int, error) {
    q, err := strconv.Atoi(qty)
    if err != nil {
        return 0, err
    }
    p, err := strconv.Atoi(price)
    if err != nil {
        return 0, err
    }
    return q * p, nil
}
```

Rust はこの定型を `?` にまとめます。`Ok` なら中身を取り出して先へ進み、`Err` ならその場で `Err` を返して関数を抜けます。

```rust
// Rust
use std::num::ParseIntError;

fn subtotal(qty: &str, price: &str) -> Result<i32, ParseIntError> {
    let q = qty.parse::<i32>()?;   // 数字でなければ、ここで Err を返す
    let p = price.parse::<i32>()?;
    Ok(q * p)
}

fn main() {
    match subtotal("3", "500") {
        Ok(total) => println!("小計 {total} 円"),      // 小計 1500 円
        Err(_) => println!("個数と単価は数字で入力してください"),
    }
}
```

`?` を使うには、関数自身が `Result` を返している必要があります。`Err` をその場で返す先が要るからです。呼び出し元の `main` は、返ってきた `Result` を `match` で受け止め、成功なら小計を、失敗なら入力し直しを促します。Go の二段の `if err != nil` が、Rust では `?` 二つに畳まれました。

この `?` は `Option` に対しても同じ形で使え、`Option` なら `None` を、`Result` なら `Err` を、それぞれ呼び出し元へ渡します。「無いかもしれない」と「失敗したかもしれない」を、同じ書き味で扱えるようにしてあるわけです。

ここまでで、Go の `(値, err)` が Rust では `Result` と `?` になり、成否を確かめないまま中の値へ進めない形に徹底されることを見ました。次は、Go の interface にあたる仕組み、共通の振る舞いを表す trait を見ます。
