# ツールチェーンとプロジェクト

Go では `go` コマンド一本で、ビルドもテストも依存管理も済ませてきたはずです。Rust もこの「一つの道具がまとめて面倒を見る」という感覚は同じで、その役割を担うのが cargo です。最初に全体像を押さえておきましょう。

Rust のツールチェーンは大きく二段構えになっています。下の層が rustup で、これはコンパイラ（`rustc`）や標準ライブラリといったツールチェーン本体のバージョンを管理します。Go でいえば、複数の Go バージョンを切り替える仕組みが標準で組み込まれている、というイメージです。その上で日々触ることになるのが cargo です。cargo はビルド、依存解決、テスト、そしてパッケージの公開までを一手に引き受けます。

`go` コマンドとの一番はっきりした違いは、cargo が crates.io というパッケージレジストリと一体で動くことです。ここは Go と感覚が違うので、少し丁寧に見ておきます。

Go では、依存を `github.com/google/uuid` のようにリポジトリのパスそのもので指定してきたはずです。パッケージの名前が、そのまま「どこにあるか」を表しています。名前が在りかを兼ねているので、名前をまとめて登録しておく中央の場所のようなものは要りませんでした。

Rust はここが違います。依存は `uuid` や `serde` のような短い名前で指定します。この名前は crates.io に登録されたもので、世界で一つしかありません。在りかを書いていないのに取ってこられるのは、cargo がその名前を crates.io に問い合わせ、対応する crate を見つけてくれるからです。crates.io が名前と実体を結びつける登録所として働いている、と考えてください。

依存の具体的な足し方はあとの「依存とビルド」で見ます。ここではまず「cargo は crates.io を前提に動く」という土台だけ押さえておけば十分です。

## cargo と crate

cargo を理解するうえで先に押さえたいのが crate という単位です。crate は、Rust においてコンパイルと配布のひとまとまりを指します。Go の感覚だと「ビルドの対象」と「配布されるもの」がここで一つの言葉に束ねられている、と捉えてください。

crate には大きく二種類あります。一つは binary crate で、`main` 関数を入口に持ち、実行可能ファイルになります。もう一つは library crate で、こちらは単体では動かず、ほかの crate から使われるための機能を提供します。crates.io に公開されてみんなが依存するのは、たいていこの library crate です。`cargo new` で作るアプリは binary crate になります。誰かが公開している便利な機能を取り込むときは、その library crate を依存に加えます。

ここで「crate と Go の package はどう対応するのか」「Rust の `mod` は何なのか」が気になるかもしれませんが、その粒度の話はこの章ではあえて踏み込みません。コードを分割するモジュールの仕組みは独立した話題で、[モジュール](modules.md)の章でまとめて扱います。この章では「crate はコンパイル・配布の単位で、crates.io から取ってこられるもの」という理解で十分です。

## プロジェクトの作り方

新しいプロジェクトは `cargo new` で作ります。

```sh
cargo new hello
```

これを実行すると、次のような構成が生成されます。

```text
hello/
├── Cargo.toml
└── src/
    └── main.rs
```

`Cargo.toml` がプロジェクトの設定ファイルで、Go の `go.mod` にあたります。中身を見るとこうなっています。

```toml
[package]
name = "hello"
version = "0.1.0"
edition = "2024"

[dependencies]
```

`[package]` セクションにプロジェクト自身の情報が書かれ、依存はこのあと `[dependencies]` に並んでいきます。`edition` は Rust の言語仕様の版を指す項目で、数年ごとに更新されます。新しく作ったプロジェクトには、その時点の最新版が自動で入ります（ここでは 2024 版）。古いコードを壊さずに言語を進化させるための仕組みなので、今は深く気にせず、生成されたままにしておけば大丈夫です。

`src/main.rs` には、すでに動く Hello World が入っています。

```rust
fn main() {
    println!("Hello, world!");
}
```

`fn main()` が実行の入口で、Go の `func main()` と同じ役割です。`println!` の末尾の `!` は、これが関数ではなくマクロであることを示しています。今は「画面に文字列を出すお決まりの書き方」と捉えておけば十分です。このコードは [Rust Playground](https://play.rust-lang.org/) に貼り付ければ、手元に何も入れなくてもブラウザ上で動かせます。

作ったプロジェクトは、ディレクトリの中で次のコマンドを使います。

```sh
cargo run     # ビルドして実行する
cargo build   # ビルドだけする
cargo test    # テストを実行する
```

`cargo run` がビルドと実行をまとめてやってくれるので、開発中はこれを叩くことが多くなります。`go run` と `go build`、`go test` がそれぞれ対応していると思えば、操作の感覚はほぼそのまま移せます。

## 依存とビルド

外部の crate に依存したくなったら、`Cargo.toml` の `[dependencies]` に書き加えます。たとえば乱数を扱う `rand` を使うなら、こう書きます。

```toml
[dependencies]
rand = "0.8"
```

ただ、手で書くよりも `cargo add` を使うほうが楽で確実です。

```sh
cargo add rand
```

こうすると、その時点で取得できる適切なバージョンを調べて `Cargo.toml` に書き込んでくれます。あとは `cargo build` や `cargo run` のときに、cargo が crates.io から依存を取得してビルドに含めます。`go get` で依存を足していく流れに近いものです。

ここで一つ、Go の経験があると誤読しやすいのがバージョンの書き方です。`rand = "0.8"` という指定は「0.8 ちょうどに固定する」という意味ではありません。これは semver にもとづく範囲指定で、既定では「0.8 と互換性のある範囲（具体的には 0.8.x）なら新しいものに上げてよい」という意味になります。つまりバージョンを一点に釘付けにしているのではなく、許容する幅を宣言しているわけです。

では実際にビルドに使われるバージョンはどこで決まるのか、というと、`Cargo.lock` です。初回のビルド時に、cargo は `Cargo.toml` の範囲指定をもとに依存グラフ全体を解決し、その結果（どの crate のどのバージョンを使うか）を `Cargo.lock` に固定して書き出します。以降は、明示的に更新しない限りこのロックされたバージョンが使われるので、誰がいつビルドしても同じ結果が得られます。

この `Cargo.lock` は、Go の `go.sum` とは役割が違うので注意してください。`go.sum` が持つのはチェックサムによる検証情報だけで、どのバージョンを使うかを決めているわけではありません。Go は、各依存が要求する最小バージョンの中から必要十分なものを選ぶ MVS という解決方式と `go.sum` の検証を組み合わせて再現性を担保しています。一方 Rust の `Cargo.lock` は、解決済みのバージョングラフそのものを丸ごと固定し（チェックサムも併せて持ちます）、それによって再現ビルドを成立させます。仕組みが異なるため、`go.sum` に「ちょうど対応するもの」は Rust にはない、と捉えるのが正確です。

運用の目安も Go と同じ感覚で大丈夫です。実行ファイルを作る binary crate のプロジェクトでは `Cargo.lock` をバージョン管理に含め、誰の環境でも同じ依存でビルドできるようにしておきます。ツールチェーンからプロジェクト、依存まで、ひと通りの足場が整いました。
