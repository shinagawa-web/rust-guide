# 速さを比べる（補足）

これは本編とは別の補足です。ここまでは Rust と JavaScript の境界を主役にしてきましたが、1章で触れた「重い処理では wasm のほうが速い」という話を、作った差分ツールで自分の手で計ってみます。

## 計測ページを開く

サンプルには、速さを比べるためのページ `bench.html` が入っています。同じ差分を、Rust から作った wasm 版と、それと同じ計算を JavaScript で書いた版の両方で走らせ、かかった時間を並べて見せます。

開く前に、wasm ができていることを確かめます。まだ作っていなければ、ここで作ります。

```sh
$ wasm-pack build --target web
```

あとは本編と同じで、ローカルサーバーを立てて開きます。

```sh
$ python3 -m http.server 8000
```

ブラウザで <http://localhost:8000/bench.html> を開きます。文字数を選んで「計測する」を押すと、結果が表で出ます。

## 何を計っているか

速さを正しく比べるには、wasm 版と JavaScript 版が同じ処理をしていないと意味がありません。そこでサンプルには、Rust の `diff` と同じアルゴリズムを JavaScript で書いた `diff.js` が入っています。中身の作りは5章で読んだ `lib.rs` とほぼ同じで、表を埋めて共通部分を探し、消えた・増えたを `[-…-]` と `[+…+]` で囲んで返します。文字単位で処理するところも同じです。

`bench.js` は、この2つを読み込んで呼び分けています。

```js
import init, { diff as wasmDiff } from "./pkg/textdiff.js";
import { jsDiff } from "./diff.js";
```

`wasmDiff` が Rust から作った wasm 版、`jsDiff` が JavaScript 版です。同じ入力を両方に渡して、かかった時間だけを比べます。

## どう計っているか

計測に使う文章は、その場で作ります。

```js
const a = makeText(n, 1);
const b = mutate(makeText(n, 2));
```

`makeText` は、指定した文字数の文章を作ります。使う文字は英字と空白だけなので、1文字が1バイトになり、wasm 版と JavaScript 版がまったく同じ量の計算をします。ここが、速さを公平に比べられる理由です。`mutate` は、その文章を少しだけ書き換えて、差分の出る相手を作ります。毎回同じ文章になるようにしているので、計測するたびに条件が揃います。

時間を計るのはこの部分です。

```js
function measure(fn) {
  for (let i = 0; i < 3; i++) fn();
  const runs = 3;
  const start = performance.now();
  for (let i = 0; i < runs; i++) fn();
  return (performance.now() - start) / runs;
}
```

最初の `for` で、本番の計測に入れずに3回だけ空回ししています。最初の数回はコンパイルなどで遅く出るので、その分を捨てる準備運動です。そのあと3回走らせ、`performance.now()` で測った時間を平均します。`performance.now()` は、ミリ秒を細かく測れる時計です。この `measure` を wasm 版と JavaScript 版のそれぞれで呼んで、結果を表にしています。

## 結果を読む

文字数を 3000 のままで計測すると、たとえば次のように出ます。

| 実装 | 時間 (ms) |
|---|---|
| wasm (Rust) | 12.4 |
| JavaScript | 24.8 |

wasm 版は JavaScript 版の約 2.0 倍の速さでした。この数字は使っているパソコンやブラウザで変わりますが、wasm のほうが速く、だいたい2倍前後になるはずです。

## なぜこれくらい速いのか

差分の計算は、2つの文章の長さをかけ合わせたぶんの大きさの表を埋める、計算量の多い処理です。こういう重い計算では、Rust から作った wasm のほうが速く動きます。Rust は書いたコードがほぼそのまま機械語になり、実行中に裏でメモリを片づける仕組み（ガベージコレクタ）も持たないぶん、速く動くからです。

ただ、速いといっても2倍くらいで、100倍のような差ではありません。1章で見たとおり、軽い処理では、境界で文字列を受け渡す手間のほうが上回って、かえって遅くなることもあります。wasm が効くのは、この差分計算のような重い処理のときです。

## 文字数を増やすと

文字数を増やして計測すると、wasm 版も JavaScript 版もかかる時間が延びます。ですが2つの倍率はだいたい変わらず、そのぶん、実際にかかる時間の差（ミリ秒）は開いていきます。計算量が増えるほど、wasm の速さが効いてくるのが見えます。

これで、1章で「重い処理では wasm のほうが速い」と言っていたことを、自分の手で計って確かめられました。速さは wasm の魅力の一つです。ただ、この初級編でいちばん見てほしかったのは、Rust で書いた関数がブラウザの中で動き、JavaScript から呼ばれる、その境界の手応えのほうです。
