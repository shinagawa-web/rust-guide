# WebAssembly とは / なぜ Rust

普段ブラウザの中で動くコードといえば JavaScript です。WebAssembly は、そこに Rust のような別の言語で書いたコードを持ち込むための仕組みです。この初級編では、その入口を実際に手を動かして体験します。まずは、そもそも WebAssembly とは何で、なぜ Rust と組み合わせるのかを見ておきます。

## WebAssembly とは

WebAssembly（短く wasm とも呼びます）は、ブラウザの中で速く動く、特定の言語に縛られない実行形式です。ブラウザが直接実行できるのは長いあいだ JavaScript だけでしたが、そこに wasm という共通の形式が加わりました。Rust をこの形式にコンパイルすれば、書いたコードをブラウザの中で動かせます。

大事なのは、コードを書く言語と、実行する場所が切り離されたことです。人が読み書きするのは Rust、ブラウザが実行するのは Rust から作った wasm、という関係になります。JavaScript を置き換えるものではなく、JavaScript の隣で一緒に動く、と考えるのがちょうどいい距離感です。

## ブラウザの中で ― JavaScript が苦手なことを任せる

ブラウザの中で動くコードは、これまで通り JavaScript で書けます。画面のボタンや入力欄を扱うような普段の処理なら、JavaScript で十分で、wasm を持ち出す必要はありません。

wasm が効いてくるのは、JavaScript には重い処理をブラウザの中でやりたいときです。画像や音声の加工、暗号の計算、ゲームのような、計算量の多い処理がそれにあたります。こうした処理は Rust で書いて wasm にすると、JavaScript より速く動くことがあります。この初級で作る差分ツールも、大きなテキストどうしを比べると計算が増えるので、その仲間です。

ただし、速さはいつでも効くわけではありません。軽い処理だと、JavaScript と wasm のあいだでデータを受け渡す手間のほうが上回って、かえって遅くなることもあります。wasm は何でも速くする魔法ではなく、重い処理でこそ効く道具です。

効きどころはもう一つあります。すでに Rust で書かれた処理を、書き直さずブラウザでも使いたいときです。一度きちんと書いた計算を、パソコン向けのアプリとブラウザとで同じものを使う——その共通部分を Rust で1つ書いておき、ブラウザでは wasm として動かせます。

## ブラウザの外で ― サーバーやその周辺で

wasm のもう一つの顔は、ブラウザとは関係のないところにあります。wasm は、安全に隔離できて、OS を選ばず、速く動かせる小さな実行単位でもあります。この性質が、いまサーバーやその周辺で強く求められています。

一つは、利用者の近くで動くプログラムです。ふつう Web サービスは、どこか一か所に置いたサーバーで動きます。利用者がそこから遠いほど、通信の往復に時間がかかって表示が遅くなります。そこで、世界中に散らばったサーバーの中から利用者に一番近い場所を選んでコードを動かし、この待ち時間を縮める、という仕組みがあります。

そこで走らせるのは、すべてのアクセスで必ず通る短い処理です。たとえば、アクセスしてきた人の地域を見てページの言語を切り替える、ログイン済みかどうかを確かめて通すかはじくかを決める、求められたサイズに画像を変換して返す、といった処理です。こうした処理を、利用者の近くで速く、しかも他の人の処理から安全に隔離して動かしたい。ここに Rust から作った wasm が向いていて、[Cloudflare](https://developers.cloudflare.com/workers/languages/rust/) や [Fastly](https://www.fastly.com/documentation/guides/compute/rust/) といったサービスがこの形で Rust の実行を受け入れています。

もう一つは、他社の製品の中に自分のコードを埋め込む使い方です。たとえば通販プラットフォームの [Shopify](https://shopify.dev/docs/apps/build/functions) では、店ごとに違う割引や送料の計算ルールを差し込めます。店側はそのルールを Rust などで書いて wasm にして渡し、Shopify は受け取った wasm を自分のサーバーの中で安全に走らせます。本体のコードを店に触らせることなく、店ごとの処理だけを安全に受け入れられるわけです。

## 安全に隔離される、という強み

外の世界のどちらの使い方にも、共通する状況があります。自分の基盤の上で、他人が書いたコードを走らせる、という状況です。ここでいちばん効いてくるのが、さきほど挙げた「安全に隔離できる」という性質です。

wasm は、既定では外の世界に手を出せません。ファイルやネットワーク、自分に渡された分の外のメモリ、OS の機能——どれも、動かす側が開けた入り口からしか触れられません。できることが最初から狭く区切られているので、預かったコードがサーバーを壊したり、隣で動く別の利用者のデータを覗いたりすることができません。

得をするのは、まず他人のコードを預かって動かすプラットフォームです。見ず知らずのコードでも、壊される心配をせずに受け入れられます。同じ基盤で動く利用者どうしも、互いのコードから守られます。そして、コードを書く側にとってのメリットは、その裏返しです。安全だからこそ、他社のプラットフォームが自分のコードを受け入れてくれる。生の実行ファイルなら決して載せてもらえない場所に、wasm なら載せてもらえます。

この隔離の仕組みは、もともとブラウザのものです。ブラウザは、見知らぬサイトのコードを安全に走らせるために、外へ勝手に手を出させない仕組みを持っています。wasm はそこから生まれたので、同じ性質を最初から備えていて、ブラウザの外に持ち出してもそのまま効きます。

## なぜ Rust か

wasm が効く場所には、速いこと、実行環境が軽いこと、安全なこと、という条件がありました。この3つに、Rust はよく噛み合います。

まず速さと軽さです。Rust は書いたコードがほぼそのまま機械語になる言語で、実行中に裏でメモリを片づける仕組み（ガベージコレクタ）を持ちません。そのぶん速く動き、wasm にしたときに一緒に詰め込む荷物も少なくて済みます。ファイルは小さく、起動も速い。リクエストごとに使い捨てで立ち上げる、エッジのような使い方にちょうど向きます。

そして安全です。ここには2つの層があります。1つは、さきほど見た wasm 自体の隔離——外に勝手に手を出せないという、動かす側から見た安全です。もう1つは、Rust という言語そのものの安全です。Rust は、確保していないメモリを読み書きするような危険な失敗を、コンパイルの時点で弾きます。隔離された箱の中でも、その中身が自分で足を踏み外して壊れることは起きてほしくありません。Rust はこの内側の安全も引き受けます。

速い・軽い・安全がそろうことは、他の言語ではなかなか同時に成り立ちません。だからブラウザの中でも外でも、wasm の中身を書く言語として Rust がよく選ばれています。これはそのまま、あなたがこれから学ぶ言語が、こうした場所に手が届く言語だ、ということでもあります。

## この初級編で作るもの

ここからは実際に手を動かします。題材は、2つのテキストを比べて違いを表示する差分ツールです。違いを計算するところは Rust で書かれていて、それを wasm にしてブラウザの中で動かし、画面まわりの JavaScript から呼び出しています。

このルートで見てほしいのは、その呼び出しの筋です。Rust で書いた関数が wasm になり、JavaScript から呼ばれて、結果が返ってくる。この受け渡しの境目を、このガイドでは境界と呼びます。境界をまたぐとき、数値はそのまま渡せますが、文字列にはひと手間が要ります。ここがこのルートで新しく学ぶところです。

進め方は、まずコードを読み、次に日本語で差分が壊れるところを直し、最後に自分が何をどう変えたかをふり返る、という順です。速さについては、作り終えたツールを使って、最後の補足ページで自分で計れます。

では、まずプロジェクトを手元で動かすところから始めます。
