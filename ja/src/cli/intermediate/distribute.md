# ツールを配布する

ここまでで、rwc は複数のファイルやディレクトリを受け取り、表示する項目を選べるようになりました。手元でひと通り動くところまで来ています。ただ、今の rwc を動かす方法は `cargo run` だけです。プロジェクトのディレクトリまで移動して、`cargo run -- ...` と打つ。開発中はこれでいいのですが、日常の道具としては使いにくい。どこにいても `rwc ファイル名` と打つだけで動いてほしいし、同じものを他の人にも配布したくなります。

この章では、実装した rwc を配る方法を扱います。まず自分の環境に入れて、そのあと他の人へ配布するところまで進めます。完璧に仕上げてからではなく、動いた今のうちに配る、という進め方です。

## まず自分の環境に入れる

自分がどこからでも使えるようにするのが、いちばん早い配布先です。プロジェクトのディレクトリで次のコマンドを実行します。

```sh
$ cargo install --path .
```

`cargo install` は、リリース向けにビルドした実行ファイルを `~/.cargo/bin` という決まった場所にコピーするコマンドです。`--path .` は「今いるディレクトリのプロジェクトを対象にする」という指定です。

本当にコピーされたか、置き場所を覗いて確かめます。

```sh
$ ls ~/.cargo/bin/rwc
```

実行結果

```text
/Users/あなた/.cargo/bin/rwc
```

パスがそのまま表示されれば置けています。`No such file or directory` と出たら、ビルドかインストールのどこかで失敗しているので、`cargo install --path .` の出力を見直します。

`~/.cargo/bin` は、Rust を入れたときに PATH（コマンドを探す場所の一覧）へ登録されています。だから、ここに入った実行ファイルは名前だけで呼び出せます。`cargo run` と違って、プロジェクトのディレクトリにいる必要はありません。どこにいても `rwc ファイル名` で動きます。

念のため、プロジェクトの外からでも呼び出せるか確かめます。

```sh
$ cd ~
$ rwc --version
```

実行結果

```text
rwc 0.1.0
```

`rwc: command not found` と出たときは、`~/.cargo/bin` に PATH が通っていません。ターミナルを開き直すか、`source ~/.cargo/env` を実行してからもう一度試します。それでも直らなければ、Rust のインストールをやり直してください。

実行ファイルの名前が `rwc` になるのは、`Cargo.toml` の `name` から来ています。

```toml
[package]
name = "rwc"
```

コードを直したら、もう一度 `cargo install --path .` を実行すれば、入っている rwc が新しいものに置き換わります。

## 配布の準備をする

ここから先は、自分のマシンの外へ配布する話です。実装した rwc が自分の GitHub リポジトリに置いてあることを前提にします。

出発点は他人のリポジトリを clone したものなので、`git push` の送り先（origin）はそのリポジトリを指したままです。まず GitHub で自分のアカウントに空のリポジトリを作り、送り先を自分のものへ張り替えてから push します。

```sh
$ git remote set-url origin https://github.com/あなたのアカウント/rwc.git
$ git push -u origin main
```

送り先を張り替えられたら、配る準備は完了です。

## 実行ファイルを配る

他の人に配るときは、ビルド済みの実行ファイルそのものを配布します。相手は Rust を用意する必要はありません。ソースからビルドしてもらうのではなく、できあがったファイルを受け取って置くだけ、という形です。

まず、配るための実行ファイルを用意します。ここで初めて `cargo build` を使います。

これまで動かすのに使ってきた `cargo run` は、ソースコードを実行ファイルに変換（ビルド）してから、それを走らせるコマンドでした。配るのに要るのは前半のビルドだけです。走らせずに止めて、できあがった実行ファイルそのものを取り出せばいい。それをやるのが `cargo build` です。

```sh
$ cargo build --release
```

`cargo build` は、ソースコードをビルドして実行ファイルを作るところまでをやるコマンドです。`--release` を付けると、最適化した実行ファイルが `target/release/rwc` にできます。`cargo run` が使う開発用のビルドより動作が速く、配るのはこちらです。

このファイルを GitHub の Release に添付します。リポジトリの Releases から新しいリリースを作り、`target/release/rwc` をアップロードします。受け取る人は、その1ファイルをダウンロードして、実行できるようにするだけです。

```sh
$ chmod +x rwc
$ ./rwc ファイル名
```

`chmod +x` は、ダウンロードしたファイルを実行できる状態にするコマンドです。

macOS では、ブラウザでダウンロードした実行ファイルに安全確認のための印が付き、最初の実行がブロックされることがあります。その場合は `xattr -d com.apple.quarantine rwc` で印を外すか、システム設定の「プライバシーとセキュリティ」から実行を許可します。

一つ注意があります。ビルドした実行ファイルは、ビルドした環境と同じ種類の OS でしか動きません。macOS で作ったものは macOS 向け、というように分かれ、同じ OS でも CPU の種類が違うと動かないことがあります。複数の環境向けをまとめて用意する方法もありますが、この章では踏み込みません。まずは自分と同じ環境の人に配布するところから始めれば十分です。

## 動かして確かめる

この章を終えた rwc は、`cargo run` を離れて、道具として配れるようになりました。自分の環境に入れれば、どこからでも呼び出せます。

```sh
$ cargo install --path .
$ rwc sample/hello.txt sample/japanese.txt
sample/hello.txt  lines: 3  chars: 16  bytes: 16
sample/japanese.txt  lines: 2  chars: 9  bytes: 23
```

配れる形にはなりましたが、実際に配布すると、こちらが想定していなかった使われ方をされます。

- 存在しないファイルを指定される
- 中身が読めないファイルを渡される

今の rwc はこうした場面の応対がまだ場当たり的です。たとえば複数渡したうちの一部が読めなくても、成功扱いのまま終わってしまうなど、扱いとして物足りない箇所が残っています。次の章では、エラー処理を本格的に整え、問題が起きたときは原因を明確に伝え、失敗を成功と取り違えずに終了するようにしていきます。
