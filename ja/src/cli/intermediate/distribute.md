# 作ったツールを人に渡す

ここまでで、rwc は複数のファイルやディレクトリを受け取り、表示する項目を選べるようになりました。手元でひと通り動くところまで来ています。ただ、今の rwc を動かす方法は `cargo run` だけです。プロジェクトのディレクトリまで移動して、`cargo run -- ...` と打つ。開発中はこれでいいのですが、日常の道具としては使いにくい。どこにいても `rwc ファイル名` と打つだけで動いてほしいし、同じものを他の人にも渡したくなります。

この章では、育てた rwc を配る方法を扱います。まず自分の環境に入れて、そのあと他の人へ渡すところまで進めます。完璧に仕上げてからではなく、動いた今のうちに配る、という進め方です。

## まず自分の環境に入れる

自分がどこからでも使えるようにするのが、いちばん早い配り先です。プロジェクトのディレクトリで次のコマンドを実行します。

```sh
$ cargo install --path .
```

`cargo install` は、リリース向けにビルドした実行ファイルを `~/.cargo/bin` という決まった場所にコピーするコマンドです。`--path .` は「今いるディレクトリのプロジェクトを対象にする」という指定です。

`~/.cargo/bin` は、Rust を入れたときに PATH（コマンドを探す場所の一覧）へ登録されています。だから、ここに入った実行ファイルは名前だけで呼び出せます。`cargo run` と違って、プロジェクトのディレクトリにいる必要はありません。どこにいても `rwc ファイル名` で動きます。

実行ファイルの名前が `rwc` になるのは、`Cargo.toml` の `name` から来ています。

```toml
[package]
name = "rwc"
```

コードを直したら、もう一度 `cargo install --path .` を実行すれば、入っている rwc が新しいものに置き換わります。

## 他の人に渡す準備をする

ここから先は、自分のマシンの外へ渡す話です。どの渡し方でも、育てた rwc が自分の GitHub リポジトリに置いてあることを前提にします。

出発点は他人のリポジトリを clone したものなので、`git push` の送り先（origin）はそのリポジトリを指したままです。まず GitHub で自分のアカウントに空のリポジトリを作り、送り先を自分のものへ張り替えてから push します。

```sh
$ git remote set-url origin https://github.com/あなたのアカウント/rwc.git
$ git push -u origin main
```

以降の渡し方は、相手が Rust を持っているかどうかで変わります。

## Rust を使う人に渡す

相手も Rust を入れているなら、リポジトリの URL を伝えるだけで済みます。相手は次のコマンドを実行します。

```sh
$ cargo install --git https://github.com/あなたのアカウント/rwc.git
```

`--git` は「このリポジトリを取ってきてビルドし、そのまま入れる」という指定です。さっき自分の環境で使った `cargo install` と同じ仕組みで、対象が手元のディレクトリからリモートのリポジトリに変わっただけです。相手の手元に rwc がインストールされ、`rwc` として使えるようになります。

## Rust が無くても使えるようにする

相手が Rust を持っていない場合は、ビルド済みの実行ファイルそのものを渡します。ソースからビルドしてもらうのではなく、できあがったファイルを受け取って置くだけ、という形です。

まずリリース向けにビルドします。

```sh
$ cargo build --release
```

`--release` を付けると、最適化した実行ファイルが `target/release/rwc` にできます。`cargo run` が使う開発用のビルドより動作が速く、配るのはこちらです。

このファイルを GitHub の Release に添付します。リポジトリの Releases から新しいリリースを作り、`target/release/rwc` をアップロードします。受け取る人は、その1ファイルをダウンロードして、実行できるようにするだけです。

```sh
$ chmod +x rwc
$ ./rwc ファイル名
```

`chmod +x` は、ダウンロードしたファイルを実行できる状態にするコマンドです。

macOS では、ブラウザでダウンロードした実行ファイルに安全確認のための印が付き、最初の実行がブロックされることがあります。その場合は `xattr -d com.apple.quarantine rwc` で印を外すか、システム設定の「プライバシーとセキュリティ」から実行を許可します。

一つ注意があります。ビルドした実行ファイルは、ビルドした環境と同じ種類の OS でしか動きません。macOS で作ったものは macOS 向け、というように分かれ、同じ OS でも CPU の種類が違うと動かないことがあります。複数の環境向けをまとめて用意する方法は中級の範囲を越えるので、上級に譲ります。まずは自分と同じ環境の人に渡すところから始めれば十分です。

## 動かして確かめる

この章を終えた rwc は、`cargo run` を離れて、道具として配れるようになりました。自分の環境に入れれば、どこからでも呼び出せます。

```sh
$ cargo install --path .
$ rwc sample/hello.txt sample/japanese.txt
sample/hello.txt  lines: 3  chars: 16  bytes: 16
sample/japanese.txt  lines: 2  chars: 9  bytes: 23
```

配れる形にはなりましたが、人に渡せば、こちらが想定していなかった使われ方をされます。存在しないファイルを指定される、中身が読めない。今の rwc はそうした場面の応対がまだ場当たり的で、書き方によっては途中で異常終了してしまう箇所も残っています。次の章では、エラーの扱いを本格的に整えて、何が起きても rwc が理由を告げて落ち着いて終われるようにしていきます。
