# 変更をふり返る

日本語ファイルでも `chars` が正しく数えられるようになり、それを守るテストも足せました。ここまでで手を入れたのは `counter.rs` の 2 か所だけです。最後に、自分が何をどう変えたのかを見返し、自分の言葉でまとめます。変更を自分の言葉で言えるようになることが、この初級編のねらいです。

## 変更を見返す

まず、自分の変更を `git diff` で見返します。`git diff` は、いま手元で変えた内容を、変えた行だけ並べて見せてくれます。

```sh
$ git diff
```

```diff
--- a/src/counter.rs
+++ b/src/counter.rs
@@ -7,7 +7,7 @@ pub struct Counter {
 impl Counter {
     pub fn count(content: &str) -> Counter {
         let lines = content.lines().count();
-        let chars = content.len();
+        let chars = content.chars().count();
         let bytes = content.len();
         Counter { lines, chars, bytes }
     }
@@ -26,4 +26,12 @@ mod tests {
         assert_eq!(c.bytes, 12);
     }
 
+    #[test]
+    fn counts_multibyte_chars() {
+        let content = "こんにちは\n世界\n";
+        let c = Counter::count(content);
+        assert_eq!(c.lines, 2);
+        assert_eq!(c.chars, 9);
+        assert_eq!(c.bytes, 23);
+    }
 }
```

`-` で始まる行が消したもの、`+` で始まる行が足したものです。変えたのは 2 か所——`chars` を数える 1 行と、末尾に足した日本語のテスト——だけだと確かめられます。

なお、確認に使った `sample/japanese.txt` は自分で作った動作確認用のファイルで、修正そのものではありません。`git status` には未追跡ファイルとして出てきますが、コードの変更は `counter.rs` の 2 か所だけです。

## 何をどう変えたか、文章にする

diff を見ながら、変更を自分の言葉で説明してみます。次の 4 つが書ければ十分です。

- 直したもの: 日本語のファイルを渡すと、`chars` が文字数ではなくバイト数になっていた。
- 原因: `chars` を `content.len()` で数えていて、`len()` は文字数ではなくバイト数を返すため、1 文字が複数バイトの日本語ではずれていた。
- 直し方: `content.chars().count()` に変え、中身を 1 文字ずつに区切ってから数えるようにした。
- 確認: 既存のテストは通ったまま、日本語のテストを足して、あとで数え方が元に戻っても気づけるようにした。

コードは読めば追えますが、「なぜその変更で直るのか」を言葉にできるかどうかは別の力です。ここまで一段ずつ読み解いてきたぶん、原因も直し方も、文字とバイトの違いのレベルで説明できるはずです。

この 4 つがそのまま、変更をレビューしてもらうために出す PR の説明文になります。何を・なぜ・どう変えたかが書いてあれば、読む人は diff を一行ずつ追わなくても変更の意図をつかめます。

## ふり返り

ここまでで、`cargo run` でツールを動かし、`main.rs` が引数を受け取ってファイルを読み、`counter.rs` が行数・文字数・バイト数を数える流れを読み、文字数がずれる原因を文字とバイトの違いから突き止め、`chars().count()` で直し、テストで守り、変更を自分の言葉でまとめました。

やったのは小さな 1 行の修正ですが、その 1 行にたどり着くために、文字列がバイトの並びでできていること、その並び方を決める UTF-8 では文字ごとにバイト数が違うこと、という一段下の仕組みを読み解いてきました。既存のコードを読んで、必要なところだけ直せる——この進め方は、題材が文字数の集計でなくなっても同じように使えます。
