# Rust Playground を使い込む

このページは、[Rust Playground の使い方](playground.md) の基本に慣れたあと、もう一歩進んで使いたくなったときのための内容です。外部クレートを使う例を動かしたい、エディションを切り替えたい、コンパイラが内部で何をしているかを覗いてみたい——そんなときに、必要なところだけ読み返してください。最初から順に読む必要はありません。

## クレートとエディションを合わせる

このガイドの例の中には、標準ライブラリだけでは完結しないものもあります。Playground には、ダウンロード数上位 100 個のクレート（Rust では、ライブラリやパッケージの単位をクレートと呼びます）と Rust Cookbook 掲載のクレート、そしてそれらの依存先があらかじめ用意されていて、コード中で `use rand::Rng;` のように参照すればそのまま使えます。乱数の `rand`、シリアライズの `serde`、非同期ランタイムの `tokio` といった、よく名前の挙がるものはたいてい含まれています。

一方で、ここに入っていない任意のクレートを後から取り寄せたり、バージョンを細かく指定したりはできません。これは後で触れるネットワークの制約とつながっています。手元の Cargo プロジェクトのように `Cargo.toml` を編集して依存を足す、という使い方は Playground では想定されていないと考えてください。

エディション（Rust の言語仕様の版で、2015 年版から数年ごとに更新されています）も結果を左右します。ツールバーの設定メニューから、2015・2018・2021・2024 のいずれかを選べます。エディションが違うと一部の文法や既定の挙動が変わります。特定のエディションを前提にした例がうまく動かないときは、ここを本文の説明に合わせてみてください。同じ設定では、コンパイラのチャンネル（stable / beta / nightly）と、ビルドモード（debug / release）も切り替えられます。新しい言語機能を試す例や、最適化後の挙動を見たい場面で効いてきます。

## できないこと、そしてその理由

Playground で「動かない」とつまずく多くは、バグではなく設計上の制約です。理由を知っておくと、詰まったときに自分のコードを疑い続けずに済みます。

最大の制約はネットワークです。実行されるコードは外の世界から切り離されたコンテナの中で動き、外部との通信路がありません。これはセキュリティのための隔離で、見知らぬ他人のコードを共有サーバーで実行する以上、外へ繋がせないのは当然の前提です。そのため、HTTP リクエストを投げる、URL からデータを取りに行くといった例は、コードが正しくても通信の段階で失敗します。

次に、扱えるのは一枚のエディタに書いたコードだけです。`mod` でモジュールに分けるところまでは一つのファイル内でできますが、複数のソースファイルに分割したプロジェクトをそのまま持ち込むことはできません。このガイドの例はこの制約に収まる範囲で書いていますが、自分のプロジェクトを丸ごと貼り付けて動かす用途には向きません。

そして、実行は使い捨てです。コンパイルと実行にかかる時間も、使えるメモリも、共有環境を守るために上限が設けられています。重い無限ループや巨大な確保は途中で打ち切られることがあります。書いたコードやその場で作ったファイルは実行が終われば消え、次に開いたときには残っていません。なお、ファイルへの読み書きそのものを試す例はありますが、それはこの一時的な環境の中だけの話で、結果を持ち帰る手段ではない、と捉えておくのが安全です。

## もう一歩踏み込んだ使い方

ツールバーには、コードを別の角度から覗くためのツールがいくつか用意されています。学習が進んで「このコードは内部でどう扱われているのか」を知りたくなったときに役立ちます。

身近なところでは、Clippy でより踏み込んだ書き方の指摘を受けられ、フォーマット機能（rustfmt）でコードを標準的な体裁に整えられます。これらは安定版でそのまま使えます。さらに踏み込むと、生成されるアセンブリや LLVM IR、コンパイラ内部表現である MIR を表示させることもできます。マクロが何に展開されているかを見る機能や、未定義動作を検出する Miri といったツールもあり、こちらは選んでいるチャンネルに関わらず自動的に nightly で実行されるので、自分で切り替える必要はありません。最初のうちは無理に触る必要はありません。「なぜそう動くのか」を本文だけでは追い切れなくなったときに、こうした覗き口があることを思い出してください。
