# ライフタイム

「参照は、指す値より長く生きられない」。参照を使うなら必ずついてくる決まりです。参照が指している先が片付けられたあとも参照だけが残っていたら、もう無い場所を読むことになる。だから Rust は、参照が指す値より長生きしていないかをコンパイル時に確かめて、危ういものを止めます。

このページであつかうのは、その「どうやって確かめているのか」です。ふだんはコンパイラが自分で確かめてくれるので、こちらは何も書きません。ただ、ときどき寿命の対応を自分で示さないと通らない場面があり、そこで `'a` という記法を書くことになります。多くの人が身構えるのはこの記号ですが、やっているのは寿命を設定することではなく、参照どうしの寿命の関係を書くだけです。その「自分で示す場面」が何で、なぜ要るのかがわかれば十分です。

## コンパイラは参照の寿命を見張っている

まず、参照が指す先より長生きすると何が起きるかを見ます。

```rust
fn main() {
    let r;
    {
        let s = String::from("hello");
        r = &s;          // s を借りる
    }                    // ここで s は片付けられ、"hello" も解放される

    println!("{r}");     // コンパイルエラー：r は、もう無い s を指している
}
```

内側のブロックを抜けた時点で `s` は片付けられます。ところが参照の `r` は外でまだ生きていて、`s` があった場所を指したままです。このまま使えたら、解放ずみの場所を読むことになります。Rust はここを `s does not live long enough`（s は十分に長く生きていない）で止めます。

コンパイラは、`r` が生きている範囲と `s` が生きている範囲を見比べて、`r` のほうが長いと判断しています。この「どちらがどれだけ生きるか」の照合を、コンパイラはコード中のすべての参照に対してやっています。参照が生きているこの範囲のことをライフタイムと呼びます。コンパイラは常にライフタイムどうしを突き合わせて、指す先より長生きする参照がないかを見張っているわけです。

では、この止まったコードを動かしたいときはどうするか。

一つめは、`s` を内側ブロックに閉じ込めず、`r` と同じ場所に置く方法です。

```rust
fn main() {
    let s = String::from("hello");   // 参照より長く生きる場所に置く
    let r = &s;
    println!("{r}");
}
```

`s` が `r` と同じだけ生きるので、`r` の指す先が先に消えることはもうありません。

二つめは、そもそも借りるのをやめて、所有権を `r` に移す方法です。

```rust
fn main() {
    let r;
    {
        let s = String::from("hello");
        r = s;               // 借りずに所有権を受け取る
    }
    println!("{r}");         // r が持ち主なので使える
}
```

`&` を外したので、`r` は借り手ではなく持ち主になります。内側のブロックを抜けても、値は `r` とともに生きています。

どちらも、変えたのは値の生きる範囲か、借りるか所有するかのほうでした。

## 参照を返すと、寿命が決められない

コンパイラがいつも自分だけで判断できるわけではありません。決められない場面があります。その代表が、参照を返す関数です。

二つの文字列を受け取って、長いほうを返す関数を書いてみます。

```rust
fn longer(a: &str, b: &str) -> &str {
    if a.len() >= b.len() { a } else { b }
}
```

これはコンパイルできず、`missing lifetime specifier`（ライフタイムの指定が足りない）で止まります。

なぜでしょう。この関数は `a` か `b`、どちらかの参照をそのまま返します。返ってきた参照が指す先は、`a` が借りている文字列かもしれないし、`b` のほうかもしれない。呼ぶ側からは、戻り値がどちらと同じだけ生きるのかがわかりません。コンパイラは、返す参照の寿命を `a` と `b` のどちらに合わせればいいのか決められないのです。

そこで、戻り値がどちらの引数と同じだけ生きるのかを、こちらから書いて教えます。

```rust
fn longer<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() >= b.len() { a } else { b }
}
```

`a`・`b`・戻り値に同じ名前 `'a` を付けると、「この三つは同じだけ生きる」という対応がコンパイラに伝わります。`<'a>` はその名前を使うという宣言、`&'a str` はその名前を割り当てた参照です。これで戻り値は `a` と `b` に合わせて生きる、と決まり、関数は通るようになります。

## 返すのが片方だけのこともある

ここで、こう思うかもしれません。どちらが返るか分からないなら、両方に同じ寿命を割り当てる、とコンパイラが勝手に決めればいいのでは、と。`longer` についてはそのとおりで、いま書いたのがまさにそれです。

ただ、それをいつも勝手に決められると困ります。関数によっては、返すのは片方だけ、ということがあるからです。たとえば、必ず `a` を返す関数を考えます。

```rust
fn first<'a>(a: &'a str, b: &str) -> &'a str {
    a
}
```

`b` には `'a` を付けていません。戻り値は `a` の寿命にだけ結びつき、`b` は関係ないからです。だから `b` は短命でもこの関数を呼べます。もしコンパイラが「引数全部に合わせる」と決め打ちしたら、この関数でも `b` を戻り値と同じだけ生かさないと呼べなくなり、本当は通るはずの呼び出しまで弾かれてしまう。どの引数から借りているかは関数ごとに違うので、コンパイラは推測せず、書き手に書かせるわけです。

## 注釈は寿命を延ばさない

この注釈で書いているのは、具体的な長さではありません。「`'a` は3行分」のような話ではなく、複数の参照に同じ名前を付けて、これらは同じだけ生きる、という対応を書いているだけです。

そして、この対応は参照を長生きさせる魔法ではありません。むしろ逆で、同じ名前を付けた参照どうしは、いちばん短いものの寿命に合わせて制限されます。次のコードで見えます。

```rust
fn longer<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() >= b.len() { a } else { b }
}

fn main() {
    let a = String::from("hello");
    let result;
    {
        let b = String::from("hi");
        result = longer(&a, &b);   // a と b を貸す
        println!("{result}");      // ここではまだ両方生きている：OK
    }                              // b が片付けられる

    println!("{result}");          // コンパイルエラー：result は b より長生きできない
}
```

`longer(&a, &b)` を呼んだ時点で、戻り値の `result` の寿命は、`a` と `b` のうち短いほうに制限されます。`b` は内側のブロックを抜けると片付けられるので、`result` もそこまでしか使えません。ブロックの外で `result` を使おうとすると、`b` はもういない、とコンパイラが止めます。

実際に返るのは長いほうの `a` なのに、なぜ `b` に引きずられるのか。呼ぶ側は関数の中身を見ず、シグネチャに書かれた `'a` だけで判断するからです。`a` も `b` も同じ `'a` と書いてある以上、戻り値はどちらにも合わせられる前提になり、短いほう、つまり `b` に合わせて寿命を見積もります。注釈は、こうして参照どうしの寿命の対応をコンパイラに伝えているだけです。

だから、このページの最初に見た止まったコードは、`'a` を足しても直りません。あそこで必要だったのは、値の生きる範囲を変えるか、借用をやめて所有権に切り替えることでした。注釈は対応を書くだけで、寿命そのものを延ばすわけではないからです。

## 構造体が参照を持つと、借用元より長生きできない

参照は、関数の引数や戻り値だけでなく、構造体のフィールドにもなれます。

たとえば、文章の一部を借りて持っておく構造体を、素直にこう書いてみます。

```rust
struct Excerpt {
    part: &str,        // 文字列の一部を借りて持つ
}
```

これはコンパイルできず、`missing lifetime specifier` で止まります。関数のときと同じエラーです。`Excerpt` が、`part` の借りている文字列より長生きしていないか、コンパイラは確かめたいのに、その手がかりがフィールドの型に書かれていないからです。

そこで、関数のときと同じように名前を付けます。

```rust
struct Excerpt<'a> {
    part: &'a str,
}

fn main() {
    let text = String::from("hello world");
    let snippet = &text[..5];      // 先頭5文字 "hello" を借りる

    let e = Excerpt { part: snippet };
    println!("{}", e.part);
}
```

`struct Excerpt<'a>` と書き、フィールドの型を `&'a str` にすると、「`Excerpt` は、`part` が借りている文字列より長くは生きられない」という対応が結びつきます。借りた文字列が先に片付けられてしまえば、`Excerpt` は無い場所を指すことになる。だから構造体に参照を持たせるなら、借用元より長生きできない、とあらかじめ書いておくわけです。

理屈は関数のときと同じで、参照が指す先より長生きしないように、寿命の対応を書いているだけです。

## まとめ

- 参照は、指す値より長く生きられない。コンパイラは全参照のライフタイム（生きている範囲）を突き合わせ、指す先より長生きする参照を止めている。
- ふだんはコンパイラが寿命の対応を自分で追えるので、こちらは何も書かない。自分で書くのは、参照を返す関数や、参照を持つ構造体のように、コンパイラだけでは対応を決められないとき。そこで書く記法が `'a`。
- その注釈は寿命を設定するものではなく、複数の参照に同じ名前を付けて「これらは同じだけ生きる」という対応を書くだけ。長生きさせる魔法ではなく、寿命はむしろ短いほうに合わせて制限される。
