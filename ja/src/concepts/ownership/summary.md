# 所有権のまとめ

ここまでで、所有権のひととおりを見てきました。全体を一つの線でつないでおきます。

- 出発点は「プログラムが使ったメモリを、誰がいつ片付けるか」。手動で片付ける言語は速いが人任せで危うく、自動で片付ける言語（ガベージコレクション）は安全だが実行中に代償を払う。Rust は、片付けをコンパイル時に決めてしまう第三の道を選んだ。それが所有権。
- 値にはちょうど一つの所有者がいて、所有者がスコープを抜けると、その値は片付けられる。だから書き忘れも二重の片付けも起きず、実行中の見張りもいらない。
- 別の変数や関数に値を渡すと、所有権が移る（move）。渡した元は使えなくなる。ヒープを持たない小さな値（整数など）はコピーで済み、元もそのまま使える。同じ中身がもう一つ要るときは `clone`。
- 所有権を渡さずに値を貸すのが借用。`&` は読み取り専用で、同時に何個でも貸せる。`&mut` は書き換えられるが、同時に一つだけ（データ競合を防ぐ）。参照は、指す値より長く生きられない。

Rust のコンパイラが所有権まわりで出してくるエラーの多くは、「この値の片付け担当は誰か」「いまそれを借りていいか」という同じ問いの現れです。ここが読めるようになっていれば、まずは十分です。おつかれさまでした。

## もっと知りたくなったら

借用には、ここまで触れていない決まりがもう一つあります。参照は、指す値より長く生きられない、というものです。次のコードで見えます。

```rust
fn main() {
    let r;
    {
        let s = String::from("hello");
        r = &s;          // s を借りる
    }                    // ここで s はスコープを抜け、片付けられる

    println!("{r}");     // コンパイルエラー：r は、もう無い s を指している
}
```

内側のブロックを抜けた時点で、`s` は片付けられ、ヒープの "hello" も解放されます。ところが参照の `r` は、その外でまだ生きていて、`s` があった場所を指したままです。もしこのまま使えたら、すでに解放された場所を読むことになります。Rust はこれもコンパイル時に止めます（`s does not live long enough`）。

この「参照が、指す値より長生きしていないか」を、コンパイラがどうやって確かめているのか。そこに踏み込みたくなったら、[ライフタイム](../lifetime.md) が続きになります。今すぐでなくてかまいません。所有権が手になじんで、もう一歩ほしくなったときに読んでみてください。
