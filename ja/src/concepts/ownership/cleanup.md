# 誰がいつ片付けるか

プログラムが確保したメモリは、使い終わったら片付けないと、使われないまま占有され続けます。ここで避けて通れないのが、その片付けを、いつ、誰がやるのか、という問いです。

この問いに、これまでの言語は大きく2つの答えを出してきました。どちらもそれなりに機能しますが、それぞれ別の弱点を抱えています。まずその2つを見て、両方の弱点を避けるために生まれた Rust の答えへ進みます。

## 人の手で片付ける — C のやり方

C では、メモリを確保したら、使い終わったときに自分で片付けます。いつ片付けるかを完全に自分で決められるので、無駄がなく、速い。正しく書けているかぎりは、これ以上ないやり方です。

問題は、その「正しく書けているかぎり」がとても難しいことです。片付けは書き手の記憶と注意に委ねられるので、次のような取りこぼしが起きます。

- 片付け忘れる。確保したまま片付けを書き忘れると、その場所は使われないのに空きません。これが積み重なると、プログラムは動いているのに使えるメモリだけが減っていきます。
- 二度片付ける。同じ場所を二回片付けてしまうと、メモリ管理の帳尻が合わなくなり、プログラムが壊れます。
- 片付けたあとに使う。もう片付けた場所を、まだ使えると思って読み書きしてしまう。そこには別の値が入っているかもしれず、静かにおかしな結果を生みます。

やっかいなのは、これらの多くが実行してみるまで表に出ず、しかも特定の条件がそろったときだけ牙をむくことです。小さいプログラムなら注意で防げても、規模が大きくなるほど、関わる全員がすべての片付けを正しく書き続けるのは難しくなります。速さと引き換えに、正しさが人の注意任せになる。これが手動の弱点です。

## 機械に任せる — ガベージコレクション

Go・Java・Python・JavaScript などは、別の答えを取ります。片付けを書き手に任せるのをやめ、機械が実行中に「もう使われていないメモリ」を見つけて自動で回収します。この仕組みをガベージコレクションと呼びます。

書き忘れも二重の片付けも起きません。片付けの心配から解放されるので、書く側はやりたいことに集中できます。実際、多くの用途ではこれでうまくいきます。

ただし、自動の片付けはただでは実現しません。次の代償の上に成り立っています。

- 計算コストがかかる。「どのメモリがもう使われていないか」を実行中に調べること自体がひとつの仕事で、プログラム本来の処理とは別に CPU を使います。片付けが自動になったぶんの手間は、消えたのではなく、実行中に肩代わりされているだけです。
- 実行が一瞬止まる。片付けの作業をしている間、プログラムの動きが一時的に止まることがあります。やっかいなのは、それがいつ起きるかを自分ではほとんど決められないことです。素早い応答が求められる場面では、この予測しづらい停止が問題になります。
- メモリを余分に抱える。不要になったメモリはその場ですぐ回収されるわけではなく、次にガベージコレクションが走るまで確保されたまま残ります。そのぶん、実際に使っている量より多めのメモリを使うことになります。

多くのアプリではこの代償は十分に見合います。けれど、応答の速さが命だったり、使えるメモリが厳しく限られていたりする場面では、この代償が重くのしかかります。安全と引き換えに、実行中のコストを払う。これがガベージコレクションの弱点です。

## Rust の第三の道 — 所有権

2つの答えを並べると、片付けには相反する要求があることが見えてきます。C のように速さと無駄のなさを取ると、正しさが人の注意任せになる。ガベージコレクションのように安全を取ると、実行中に代償を払う。速さも安全も、と欲張ると、どちらかを諦めることになりそうに見えます。

Rust は、このどちらでもない第三の道を選びました。片付けを実行中に判断するのをやめ、コンパイル時に決めてしまうのです。

やり方はこうです。値それぞれに、片付ける担当をちょうど一つ決めておきます。そして、その担当の出番が終わった時点で、その値を片付ける。担当が誰で、出番がどこで終わるかは、コードの形から決まります。だからコンパイラは、プログラムを動かさなくても、どの値がどこで片付くのかを見通せます。

この一手で、2つの弱点が同時に消えます。片付けはコンパイラが漏れなく面倒を見るので、書き忘れも二重の片付けも起こりません。手動の危うさが無くなります。しかも片付ける場所はコンパイル時に決まっているので、実行中に見張る処理も、予測できない停止もありません。ガベージコレクションの代償も無くなります。速さはそのままに、安全だけを手に入れる。これが所有権の狙いです。
